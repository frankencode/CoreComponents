15 /** \brief %Object references
16  *
17  * The Ref template class implements object references that behave similiar to object
18  * references in higher level languages, like for instance Java. The references can
19  * be freely assigned and passed in places where a pointer to an object of compatible
20  * type is expected. In constrast to simple pointers type conversions on assignment are
21  * done automatically. Most of the time there shouldn't be a need to invoke C++'s lowlevel
22  * type casting functions manually. The only exception is if a reference is passed to an
23  * interface that would require an upcast in the type hierarchy. In this case you can
24  * use the cast<T>() helper function, which safely selects the proper low-level type cast.
25  * If the type compatiblity can only be resolved at compile time the reference will be
26  * assigned a null pointer in the same way a dynamic_cast would results in a null pointer,
27  * but knowledge of these low-level concepts is not required to use the Ref class properly.
28  * The type T needs to be inherited from the Object class, which provides the reference
29  * count for an object. The reference count starts with zero when the object is created and
30  * is incremented each time a new Ref<T> starts pointing to that object. In the same fasion
31  * the object's reference count is decremented whenever a Ref<T> stops pointing to that object.
32  * When the object's reference count reaches zero back again the object will be deleted
33  * automatically. Both reference counting and automatic destruction is guaranteed to be thread-safe.
34  * Whenever an object is created an initial reference should be created with it.
35  * Otherwise memory leaks might appear. Best practise is to use a named static constructor
36  * method and hide the C++ "constructor" (in reality the initializer) as a protected or
37  * private method. For instance see File::open(). It also results in more flexible and
38  * more readable code.
39  * The Ref template class does purposely not implement a copy-on-write (COW) strategy.
40  * This means objects, whose reference count is greater than 1, are not cloned automatically,
41  * when accessed in a modifying way. Objects have to be cloned manually and you have to be
42  * aware about the locality and thread-safety issues that arise as a consequence. For passing
43  * objects amoung threads best practise is to use the Channel class and ensure that two threads
44  * never hold a reference to the same object at the same time (shared nothing architecture).
45  * If nevertheless objects need to be shared amoung threads at the same time (for instance a singleton)
46  * make sure those objects are immutable and cannot be modified accidentally after construction.
47  * The only copy automatism the Ref template class implements is preemptive copy on assignment of a
48  * const pointer (const T *). In this case a T::copy(const T *) is called to create a copy of the
49  * of the object before assignment. This copying is needed because each Ref<T> allows modification
50  * of the referenced object.
51  *
52  * \see Singleton
53  */
